# Event-Driven Architecture

> *"Don't call us, we'll call you."* — The Hollywood Principle, and the foundation of event-driven systems.

---

## Phase 1: The Fundamentals

### What is Event-Driven Architecture (EDA)?

EDA is an architectural pattern where the flow of the program is determined by **events** — significant changes in state — rather than sequential procedure calls. Services **emit events** when something happens, and other services **react** to those events.

```
Request-Driven (Imperative):
  "Hey Tracking Service, update the status of SHP-001 to DELIVERED."
  (Direct command — caller tells receiver exactly what to do)

Event-Driven (Reactive):
  "Shipment SHP-001 has been delivered."
  (Announcement — emitter doesn't know or care who's listening)
```

### The Core Philosophy

In traditional architecture, services **tell** each other what to do.  
In event-driven architecture, services **announce** what happened — and other services decide how to react.

```
Traditional (Command-based):
  Booking Service → calls → Notification Service.sendEmail(booking)
  Booking Service → calls → Analytics Service.recordBooking(booking)
  Booking Service → calls → Billing Service.createInvoice(booking)
  
  Booking Service knows about ALL downstream services (tight coupling)

Event-Driven:
  Booking Service → emits → "BookingConfirmed" event
    → Notification Service reacts: sends email
    → Analytics Service reacts: records metric
    → Billing Service reacts: creates invoice
    → (Any new service can subscribe without changing Booking Service)
    
  Booking Service knows about ZERO downstream services (loose coupling)
```

### Events vs Commands vs Queries

| Type | Intent | Direction | Example |
|---|---|---|---|
| **Event** | "This happened" | Broadcast (1 → many) | `ShipmentDelivered` |
| **Command** | "Do this" | Targeted (1 → 1) | `SendNotification` |
| **Query** | "Tell me this" | Request-response (1 → 1) | `GetShipmentStatus` |

### Why EDA?

| Benefit | Explanation |
|---|---|
| **Loose coupling** | Producer doesn't know consumers; add/remove consumers freely |
| **Scalability** | Consumers scale independently based on their own load |
| **Resilience** | Producer succeeds even if consumer is down (events buffered in queue) |
| **Extensibility** | New features = new consumers, zero changes to existing services |
| **Real-time** | Events trigger immediate reactions (near real-time processing) |
| **Audit trail** | Events are a natural log of everything that happened |

## Phase 2: Event Types

### Domain Events

Events representing **business-significant state changes** in your domain.

```java
public class ShipmentStatusChanged {
    String shipmentId;       // "SHP-001"
    String previousStatus;   // "IN_TRANSIT"
    String newStatus;        // "DELIVERED"
    Instant occurredAt;      // 2024-01-15T14:30:00Z
    String location;         // "Chennai Port"
    String triggeredBy;      // "carrier-webhook"
}
```

**Examples in logistics:**
- `BookingCreated`, `BookingConfirmed`, `BookingCancelled`
- `ShipmentPickedUp`, `ShipmentInTransit`, `ShipmentDelivered`
- `InvoiceGenerated`, `PaymentReceived`
- `ContainerAllocated`, `ContainerReleased`
- `CustomsCleared`, `CustomsHeld`

---

### Integration Events

Events for **cross-service communication**. Simplified versions of domain events shared across service boundaries.

```java
// Internal domain model (rich, detailed)
class ShipmentAggregate {
    List<RouteSegment> segments;
    WeightBreakdown weights;
    CustomsDeclaration customs;
    // ...dozens of fields
}

// Integration event (minimal, only what consumers need)
class ShipmentDeliveredEvent {
    String shipmentId;
    String customerId;
    Instant deliveredAt;
    String deliveryLocation;
}
```

**Rule:** Integration events should be **thin** — only include data that consumers need. Don't leak your internal domain model.

---

### Change Data Capture (CDC) Events

Events generated by **capturing database changes** — every INSERT, UPDATE, DELETE becomes an event.

```
Database: shipments table
  UPDATE shipments SET status = 'DELIVERED' WHERE id = 'SHP-001'
  
CDC tool (Debezium) captures this change:
  → Publishes event to Kafka: {
      "op": "u",
      "before": { "id": "SHP-001", "status": "IN_TRANSIT" },
      "after":  { "id": "SHP-001", "status": "DELIVERED" },
      "source": { "table": "shipments", "ts_ms": 1705312200000 }
    }
```

**Tools:** Debezium (Kafka Connect), AWS DMS, Maxwell

**When to use:** When you want event-driven behavior without modifying existing application code — CDC captures events at the database level.

## Phase 3: EDA Patterns

### Pattern 1: Event Notification

The simplest form — a service emits an event saying **"something happened"**. Consumers decide what to do with it. The event carries **minimal data** — just enough to identify what happened.

```
Event: { "type": "ShipmentDelivered", "shipmentId": "SHP-001" }

Consumer receives event → calls back to producer's API for details:
  GET /api/shipments/SHP-001 → gets full shipment data
```

**Pros:** Simple events, small payload, producer API stays the single source of truth.
**Cons:** Consumer must call back → creates coupling and additional load on producer.

---

### Pattern 2: Event-Carried State Transfer

The event carries **all the data** the consumer needs — no callback required.

```
Event: {
    "type": "ShipmentDelivered",
    "shipmentId": "SHP-001",
    "customerId": "CUST-123",
    "customerEmail": "john@acme.com",
    "origin": "INMAA",
    "destination": "USLAX",
    "carrier": "MAERSK",
    "deliveredAt": "2024-01-15T14:30:00Z"
}

Consumer has everything it needs → no callback to producer.
```

**Pros:** Consumers are fully independent (no callbacks), better resilience.
**Cons:** Larger events, data duplication, consumers may have stale local copies.

**This is the most common pattern in production microservices.**

---

### Pattern 3: Event Sourcing

Instead of storing the **current state**, store the **complete sequence of events** that led to the current state. State is derived by replaying events.

```
Traditional (State-based):
  Table: shipments
  | id      | status    | carrier | location      |
  | SHP-001 | DELIVERED | MAERSK  | Chennai Port  |
  
  (Only the CURRENT state is stored. History is lost.)

Event Sourcing:
  Event Store:
  | sequence | event_type          | data                                          |
  | 1        | ShipmentCreated     | { id: "SHP-001", origin: "Mumbai" }           |
  | 2        | CarrierAssigned     | { id: "SHP-001", carrier: "MAERSK" }          |
  | 3        | ShipmentPickedUp    | { id: "SHP-001", location: "Mumbai Port" }    |
  | 4        | ShipmentInTransit   | { id: "SHP-001", location: "Arabian Sea" }    |
  | 5        | CustomsCleared      | { id: "SHP-001", port: "Chennai" }            |
  | 6        | ShipmentDelivered   | { id: "SHP-001", location: "Chennai Port" }   |
  
  Current state = replay events 1-6:
    { id: "SHP-001", status: "DELIVERED", carrier: "MAERSK", location: "Chennai Port" }
```

**Benefits:**
- **Complete audit trail** — know exactly what happened, when, and in what order
- **Time travel** — reconstruct the state at any point in time
- **Debugging** — replay events to reproduce bugs
- **New projections** — add a new consumer, replay all history to build its view
- **Natural fit** — Kafka IS an event log, making it a natural event store

**Drawbacks:**
- **Complexity** — rebuilding state from events requires careful implementation
- **Querying** — can't easily `SELECT * WHERE status = 'DELIVERED'` from an event log
- **Storage** — events accumulate forever (snapshotting mitigates this)
- **Schema evolution** — old events may have different formats than new ones

**When to use:**
- Compliance/audit requirements (logistics, finance, healthcare)
- Systems where understanding "how we got here" matters
- When you need to replay/reprocess historical data

### Pattern 4: CQRS (Command Query Responsibility Segregation)

Separate the **write model** (optimized for commands) from the **read model** (optimized for queries).

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  Write Side (Commands)          Read Side (Queries)      │
│                                                          │
│  POST /bookings ──→ Command    GET /bookings ──→ Query   │
│                     Handler                    Handler   │
│                      ↓                           ↑       │
│                  Event Store ──→ Projection ──→ Read DB   │
│                  (Kafka)        (Consumer)     (Optimized)│
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**Example:**

```
Write side:
  POST /api/bookings → validates → stores BookingCreated event in Kafka
  
  Event: { type: "BookingCreated", bookingId: "BK-001", 
           customer: "ACME", routes: [...], containers: [...] }

Projection consumer:
  Reads BookingCreated event → writes to read-optimized tables:
    - booking_summary table (for list views)
    - booking_detail table (for detail views)
    - customer_bookings table (for per-customer queries)
    - daily_booking_stats table (for dashboard)

Read side:
  GET /api/bookings?customer=ACME → queries customer_bookings table directly
  GET /api/dashboard/today → queries daily_booking_stats table directly
  (No joins, no complex queries — each read model is pre-optimized)
```

**Benefits:**
- Write and read models scale **independently**
- Read models are denormalized — blazing fast queries with no joins
- Multiple read models from same events (SQL for queries, Elasticsearch for search, Redis for real-time)
- Write model stays clean and focused on business rules

**When to use:**
- Read and write patterns are fundamentally different
- Read-heavy systems where query optimization matters
- Systems with complex business rules on the write side
- When you need multiple "views" of the same data

---

### Pattern 5: Saga (Event-Driven Orchestration)

Manage distributed transactions across services using events, as covered in [[Microservices Patterns]].

```
Choreography Saga via events:
  BookingService → publishes BookingCreated
  InventoryService → consumes BookingCreated → publishes InventoryReserved
  PaymentService → consumes InventoryReserved → publishes PaymentCompleted
  BookingService → consumes PaymentCompleted → marks booking CONFIRMED

On failure:
  PaymentService → publishes PaymentFailed
  InventoryService → consumes PaymentFailed → releases inventory (compensation)
  BookingService → consumes PaymentFailed → cancels booking (compensation)
```

## Phase 4: Event Schema Design

### Event Envelope

Every event should have a standard envelope with metadata:

```json
{
    "eventId": "evt-a1b2c3d4",
    "eventType": "ShipmentDelivered",
    "version": "1.2",
    "timestamp": "2024-01-15T14:30:00Z",
    "source": "tracking-service",
    "correlationId": "req-xyz-789",
    "causationId": "evt-previous-123",
    "data": {
        "shipmentId": "SHP-001",
        "deliveredAt": "2024-01-15T14:30:00Z",
        "location": "Chennai Port"
    }
}
```

| Field | Purpose |
|---|---|
| `eventId` | Unique identifier for idempotency |
| `eventType` | What happened (used for routing/filtering) |
| `version` | Schema version for backward compatibility |
| `timestamp` | When the event occurred |
| `source` | Which service produced the event |
| `correlationId` | Traces the original request across all services |
| `causationId` | Which event caused this event (event chain) |
| `data` | The actual event payload |

### Schema Evolution

Events are stored forever (in event sourcing). Schemas WILL change. You need backward compatibility.

**Rules:**
1. **Add fields** — always safe (old consumers ignore new fields)
2. **Remove fields** — dangerous (old events don't have new required fields)
3. **Rename fields** — breaking change (avoid)
4. **Change types** — breaking change (avoid)

**Strategy: Always Additive**

```json
// Version 1
{ "shipmentId": "SHP-001", "status": "DELIVERED" }

// Version 2 — added field (backward compatible)
{ "shipmentId": "SHP-001", "status": "DELIVERED", "deliveryProof": "SIGNED" }

// Consumer of v1 events: ignores "deliveryProof" — still works ✅
// Consumer of v2 events: uses "deliveryProof" — works ✅
```

**Tools:** Apache Avro (schema registry), Protobuf, JSON Schema

## Phase 5: Event-Driven in Spring Boot

### Using Spring Application Events (In-Process)

For events **within a single service** (no external broker needed):

```java
// Define the event
public record BookingConfirmedEvent(String bookingId, String customerId, Instant confirmedAt) {}

// Publish the event
@Service
@RequiredArgsConstructor
public class BookingService {
    private final ApplicationEventPublisher eventPublisher;

    @Transactional
    public Booking confirmBooking(String bookingId) {
        Booking booking = bookingRepo.findById(bookingId);
        booking.setStatus("CONFIRMED");
        bookingRepo.save(booking);
        
        eventPublisher.publishEvent(
            new BookingConfirmedEvent(bookingId, booking.getCustomerId(), Instant.now()));
        
        return booking;
    }
}

// Consumer 1: Send notification (async, non-blocking)
@Component
public class NotificationListener {
    @Async
    @EventListener
    public void onBookingConfirmed(BookingConfirmedEvent event) {
        emailService.sendConfirmation(event.customerId(), event.bookingId());
    }
}

// Consumer 2: Update analytics (async)
@Component
public class AnalyticsListener {
    @Async
    @EventListener
    public void onBookingConfirmed(BookingConfirmedEvent event) {
        analyticsService.recordBooking(event.bookingId());
    }
}
```

**Limitation:** Events are lost if the service crashes before consumers process them. For durability, use Kafka/RabbitMQ.

### Transactional Outbox Pattern

**Problem:** You want to save to the database AND publish an event **atomically**. But the database and message broker are two different systems — no shared transaction.

```
❌ Race condition:
  1. Save booking to DB → ✅
  2. Publish event to Kafka → ❌ (Kafka is down)
  → Booking saved but event never published → Downstream services never know

❌ Reverse order:
  1. Publish event to Kafka → ✅
  2. Save booking to DB → ❌ (DB error)
  → Event published for a booking that doesn't exist!
```

**Solution: Outbox Table**

```
1. Within a SINGLE database transaction:
   - Save the booking
   - Save the event to an "outbox" table

2. A separate process reads the outbox table and publishes to Kafka
   - If publishing fails → retry later
   - If publishing succeeds → mark as published

Database transaction guarantees booking + event are saved atomically.
```

```sql
CREATE TABLE outbox_events (
    id UUID PRIMARY KEY,
    event_type VARCHAR(100),
    payload JSONB,
    created_at TIMESTAMP DEFAULT NOW(),
    published BOOLEAN DEFAULT FALSE
);
```

```java
@Transactional
public Booking confirmBooking(String bookingId) {
    Booking booking = bookingRepo.save(updatedBooking);
    
    // Event saved in SAME transaction as booking
    outboxRepo.save(new OutboxEvent(
        UUID.randomUUID(),
        "BookingConfirmed",
        objectMapper.writeValueAsString(new BookingConfirmedEvent(bookingId)),
        false
    ));
    
    return booking;
}

// Separate scheduled job publishes events
@Scheduled(fixedDelay = 1000)
public void publishOutboxEvents() {
    List<OutboxEvent> events = outboxRepo.findByPublishedFalse();
    for (OutboxEvent event : events) {
        kafkaTemplate.send("booking-events", event.getPayload());
        event.setPublished(true);
        outboxRepo.save(event);
    }
}
```

**Even better:** Use **Debezium** to read the outbox table via CDC — no polling needed.

## Phase 6: Event-Driven Pitfalls

### Pitfall 1: Event Ordering Challenges

Events processed out of order can cause incorrect state:

```
Correct order:    ShipmentCreated → ShipmentPickedUp → ShipmentDelivered
Out of order:     ShipmentDelivered → ShipmentCreated → ShipmentPickedUp

If consumer processes "Delivered" first → tries to update a shipment that doesn't exist yet!
```

**Solution:** Use partition keys in Kafka (shipment ID) to guarantee per-entity ordering. See [[Message Queues]] Phase 10.

---

### Pitfall 2: Dual Write Problem

Writing to two systems (DB + message broker) without atomicity:

**Solution:** Transactional Outbox Pattern (Phase 5 above).

---

### Pitfall 3: Event Schema Coupling

If producers change event schema without notice, all consumers break.

**Solution:**
- Schema registry (Confluent Schema Registry)
- Backward-compatible changes only (additive)
- Version field in event envelope
- Consumer tolerance (ignore unknown fields)

---

### Pitfall 4: Eventual Consistency Confusion

In EDA, there's always a delay between when an event is produced and when all consumers have processed it.

```
T=0:   Booking confirmed → event published
T=0:   Booking Service returns "CONFIRMED" to user
T=500ms: Dashboard consumer processes event → dashboard shows new booking

User: "I just confirmed, but the dashboard doesn't show it!"
```

**Solutions:**
- Show "Your booking is confirmed. Dashboard updates in a few seconds."
- Read-your-writes: after write, query the write model directly
- Optimistic UI: update client immediately, reconcile later

---

### Pitfall 5: Debugging Distributed Events

A request flows through 8 services via events. Something went wrong. Where?

**Solution:** 
- **Correlation ID** — propagate the same ID across all events in a flow
- **Distributed tracing** — Zipkin/Jaeger (see [[Microservices Patterns]])
- **Centralized logging** — all services log events with correlation ID

```
Search logs: correlationId = "req-xyz-789"
  → Booking Service: BookingCreated at T=0
  → Inventory Service: InventoryReserved at T=100ms
  → Payment Service: PaymentFailed at T=500ms ← Found the problem!
  → Inventory Service: InventoryReleased at T=600ms (compensation)
```

## Phase 7: When EDA is an Overhead

### Scenario 1: Simple CRUD Applications

A basic admin panel with straightforward create/read/update/delete operations. Events add complexity for no benefit.

---

### Scenario 2: Synchronous User Flows

User submits a form and expects an immediate, complete response. Event-driven processing means the user might not see the result immediately.

```
User: "Create booking and show me the confirmation"
EDA: "Booking event published. Check back later for confirmation."
User: "That's a terrible UX."
```

---

### Scenario 3: Small Team, Simple System

3 services, 2 developers. The overhead of Kafka, schema registry, dead letter queues, and event monitoring outweighs the benefits.

---

### Scenario 4: Tightly Coupled Operations

If Service A **always** needs a synchronous response from Service B before it can proceed, events just add latency without enabling any decoupling.

## Phase 8: Real-World Case Studies

### Case 1: Logistics — Shipment Lifecycle Events

```
Carrier Webhook → Ingestion Service → "ShipmentStatusChanged" event → Kafka

Consumers:
  Tracking Service    → updates tracking database + customer-facing portal
  Notification Service → sends SMS/email: "Your shipment has arrived"
  Billing Service     → triggers final invoice on delivery
  Analytics Service   → updates KPI dashboard
  Compliance Service  → logs milestone for customs audit trail
  Partner API Service → pushes status to partner via webhook

Adding a new consumer (e.g., Insurance Service):
  → Subscribe to "ShipmentStatusChanged" topic
  → Process events independently
  → ZERO changes to existing services
```

---

### Case 2: Logistics — Rate Card Update Propagation

```
Admin uploads new tariff → Rate Service publishes "TariffUpdated" event

Consumers:
  15 microservices that cache rates → evict caches → reload on next request
  Audit Service → logs tariff change for compliance
  Notification Service → emails sales team about rate changes

This connects to [[Caching]] — event-based cache invalidation.
```

## Key Takeaways

1. **EDA = services announce what happened, other services react** — loose coupling at its core
2. **Event-Carried State Transfer** is the most practical pattern — events carry full data, no callbacks
3. **Event Sourcing** stores history, not just current state — powerful for audit and replay
4. **CQRS** separates reads from writes — each optimized independently
5. **Transactional Outbox** solves the dual-write problem — save event + data atomically
6. **Schema evolution must be additive** — never remove or rename fields in events
7. **Correlation IDs are essential** — without them, debugging distributed flows is impossible
8. **Eventual consistency is a feature, not a bug** — design UX around it
9. **Don't over-event** — simple CRUD and synchronous flows don't need EDA
